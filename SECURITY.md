# Security Policy

## Supported Versions

Security fixes are applied to the latest code on `main` and the most recent published release. Older releases are not maintained separately.

## Reporting a Vulnerability

Do not report security vulnerabilities in a public issue.

Use GitHub's **Report a vulnerability** option on the repository Security tab. If private vulnerability reporting is unavailable, contact the repository owner privately through their GitHub profile and ask for a secure reporting channel. Do not include secrets or personal data in the initial message.

Include only the information needed to reproduce and assess the problem:

- A concise description and expected impact.
- Affected extension version or commit.
- Reproduction steps or a minimal proof of concept.
- Relevant browser and Albert page context.
- Suggested remediation, if known.

Never submit NYU credentials, session cookies, access tokens, student records, private Albert URLs, or unredacted screenshots. Replace sensitive values with clear placeholders.

## Response Process

Maintainers will acknowledge a complete report as soon as practical, investigate its impact, and coordinate remediation and disclosure with the reporter. Please allow time for a fix before publishing details. Reports that require unsafe interaction with enrollment or other account-changing controls may be declined or require a safer reproduction.

The project does not currently offer a bug bounty or guarantee a specific response deadline.
